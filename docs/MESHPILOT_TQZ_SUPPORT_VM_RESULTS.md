# MeshPilot TQZ(XII) 支座构件族：真实 VM 结果

## 结论

基于 `通桥[2023]2322A-II` 第 34、36 个 PDF 页面尺寸表建立的 6 个 TQZ(XII) 局部三维算例，已在 Ubuntu 24.04、真实 Gmsh 和 CalculiX 2.21 上全部完成。

- reference：6/6 完成；
- uniform coarse：6/6 完成；
- 冷启动离散 PSO：6/6 可行；
- 守卫 Batch-Transfer PSO：6/6 可行；
- 每次优化均遵守 32 次真实唯一 FE 硬预算；
- 荷载合力和等效弯矩合同全部通过。

本结果是局部三维线弹性算法基准，不是支座、垫石或整孔箱梁的工程验收计算，所有结果仍为 `review_required=true`。

## CalculiX 201 的真实原因

首轮只有 1/6 通过，其余算例在读取输入文件时返回 201。完整日志表明，最先发生的致命错误不是荷载过大，也不是 PSO：

```text
*ERROR reading *NODE. Card image:
141,-1.477928890381008E-12,-570,0
```

OpenCASCADE/Gmsh 在理论坐标轴位置产生了约 `1e-12 mm` 的浮点噪声。原写法用高精度通用格式输出，形成过长的科学计数节点字段；CalculiX 2.21 没有正确读入该节点。后续出现的 `node ... is not defined` 和 `CLOAD` 错误都是节点卡读取中断后的次生错误。

最终修复没有修改几何、材料、荷载或 PSO：

1. `|coordinate| < 1e-9 mm` 的坐标写为 `0`；
2. 节点和荷载数值使用紧凑、长度受控的格式；
3. Gmsh 节点和单元标签统一映射到连续的 `1..N`、`1..E`，作为防御性兼容；
4. 节点集每行缩短，避免输入卡过长。

下表的 FE 次数只统计优化阶段新增的真实唯一 CalculiX 求解。每个算例共同需要的 reference 和 coarse 预处理不计入冷启动/迁移差值；warm probe 计入迁移次数。

| 指标 | 冷启动 PSO | 守卫 Batch-Transfer PSO | 变化 |

| 平均相对位移误差 | 0.322768% | 0.123722% | **-0.199046 个百分点** |

迁移只少了 8 次 FE，但平均目标值和误差明显更好。这说明在本批次中，历史解更主要的作用是改善有限预算下的搜索起点，而不是大幅缩减所有算例的预算。

## 业务含义

第一批真实参数化构件给出的信号是：

1. **输入稳健性先于算法比较。** Gmsh 到 CalculiX 的节点卡兼容错误足以让 5/6 算例失败，必须把求解器合同作为批量系统的一部分。
2. **迁移收益不是只有“少算”。** 本批次 FE 只减少 4.17%，但平均误差从 0.323% 降到 0.124%。
3. **门卫确实在工作。** 三个算例恢复完整预算，说明它没有把参数相近机械等同于网格答案可靠。
4. **下一批需要更多连续样本。** 目前只有 6 个点，且跨越 4500/5000 两个等级。增加同一承载等级内的连续尺寸/Ag 样本后，才适合评估 FE 降幅是否会随历史积累增加。

## 解释边界

本模型包含局部 C50 线弹性实体、名义竖向合力、`1.5AgP` 水平合力、以 H 为力臂的等效弯矩，以及六个网格控制 patch。

当前不包含：

- 支座接触和摩擦；
- 显式钢筋与预应力筋；
- 混凝土开裂、塑性和损伤；
- 31.5 m 整梁全局响应；
- 工程安全验收结论。
