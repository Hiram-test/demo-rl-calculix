# 工程师问题语料编码方法

## 目标

该语料库记录的是“工程师试图完成什么工作、遇到什么障碍、希望得到什么形式的帮助”，不是论文主题目录，也不是论坛答案汇编。

## 采样单位

- `source_group_id`：一个公开帖子、问答页或官方 FAQ；
- `question_id`：从该来源拆出的一个原子问题；
- 一个来源可以有多个原子问题；
- 不把回复数量、点赞或浏览量当作需求频率；
- 不收录无法追溯到原始页面的二手转述。

## 原子化规则

复合问题需拆开。例如：

> 为什么接触不收敛，应该改网格、摩擦还是稳定化参数？

拆成：

1. 当前不收敛最可能属于接触定义、网格还是数值控制？
2. 接触面网格应怎样调整？
3. 稳定化是否会污染目标结果，如何验证？

拆分是为了识别不同的用户工作，不是人为增加“用户人数”。统计时同时保留来源数和原子问题数。

## 核心字段

| 字段 | 含义 |
|---|---|
| `question_paraphrase` | 对公开问题的人工转述，不复制原文 |
| `explicit_need` | 用户明确说出的需求 |
| `latent_need` | 为真正解决问题仍需满足的隐含需求 |
| `need_family` | 技术问题族 |
| `engineer_job` | 工程师要完成的工作 |
| `workflow_phase` | 问题发生的工作阶段 |
| `engineering_qoi` | 用户最终关心的工程结果 |
| `failure_or_risk` | 错误处理可能造成的风险 |
| `desired_answer_type` | 用户希望获得的回答形式 |
| `answer_mode` | 归并后的答案模式 |
| `mesh_role_group` | 网格是主因、联合因素、间接因素、应先处理模型或属于工作流问题 |
| `mesh_role` | 更细的网格角色标签 |
| `review_status` | 当前人工审查状态 |

## 编码原则

### 只编码问题，不采信论坛答案

社区来源用于发现：

- 工程师怎样描述任务；
- 哪一步卡住；
- 哪些概念容易混淆；
- 需要什么检查或解释。

技术规则仍需由官方文档、标准、验证例和正式研究支持。

### 显式需求与隐含需求分开

用户可能问“单元尺寸设多少”，但隐含需求可能是：

- 先确定使用梁、壳还是实体；
- 先识别应力奇异性；
- 先定义疲劳应力提取协议；
- 先修复未连接界面。

AI不能直接回答显式问题而跳过决定答案有效性的隐含前提。

### 网格角色分级

建议使用以下语义：

- `primary`：问题直接是网格密庠、过渡、质量或收敛；
- `joint`：网格与接触、材料、时间步、连接等共同决定；
- `indirect`：网格影响结果，但主要障碍是结果语义或验证；
- `model_before_mesh`：必须先修正模型层级、边界、连接或载荷；
- `workflow`：主要是自动化、复用和批量执行。

### 不把样本频率解释成人群频率

当前公开问题样本存在：

- 英语来源偏差；
- Abaqus/Ansys 用户过度代表；
- 遇到异常的用户更可能提问；
- 成功完成的日常任务不容易留下公开记录；
- 一个帖子可拆出多个原子问题。

因此报告使用“本样本中出现多少个原子问题”，不得写成“多少比例工程师有此需求”。

## 质量检查

提交前自动检查：

- 原子问题 ID 唯一；
- 来源组与 URL 可追溯；
- 每个来源至少一个问题；
- 所有必填字段非空；
- `workflow_phase`、`engineer_job`、`mesh_role` 使用受控词表；
- 汇总 JSON 与 CSV 行数一致；
- URL 为 HTTP/HTTPS；
- 中文转述不包含长段原文。

## 后续人工复核

每一轮新增问题至少需要一名复核者检查：

1. 是否忠实反映原问题；
2. 是否被过度拆分；
3. 显式和隐含需求是否混淆；
4. QoI与风险是否有证据；
5. 分类是否应多标签；
6. 是否把软件缺陷误判为工程普遍需求。
