# 桥梁有限元工程师问题语料：第一轮发现

## 这次调研的对象

此前的 100 条案例/论文/官方资料主要回答“有哪些方法和研究边界”，不能直接代表工程师日常会问什么。

本轮只收集**公开可追溯的工程问题**，并把一个帖子或 FAQ 中的复合问题拆成原子问题：

- 81 个独立帖子或官方 FAQ；
- 174 个原子问题；
- 141 个软件社区问题、25 个科研社区问题、8 个官方 FAQ；
- 57 个 SIMULIA Community 问题、57 个 Ansys Innovation Space 问题、25 个 ResearchGate 问题，另含 Eng-Tips、Dlubal、SOFiSTiK、Autodesk 与 Physics Forums；
- 所有问题均人工转述并编码；论坛回复不作为权威技术结论。

## 第一发现：多数工程师问题发生在“网格之前”

| 工作阶段 | 原子问题数 | 样本占比 |
|---|---:|---:|
| 问题定义与建模 | 84 | 48.3% |
| 验证与验收 | 37 | 21.3% |
| 网格生成与布置 | 25 | 14.4% |
| 结果提取与解释 | 11 | 6.3% |
| 批量与工作流 | 9 | 5.2% |
| 求解过程 | 8 | 4.6% |

这意味着在本样本中，工程师最常卡住的并不是“单元尺寸填多少”，而是：

1. 这个有限元模型能不能回答我要的问题；
2. 构件、支座、钢筋、接触和约束是否正确传力；
3. 结果为什么与试验、规范或另一个模型不一致；
4. 什么时候可以相信结果并停止修改模型。

该结论只描述当前公开问题样本，不能外推为全球桥梁工程师的统计比例。

## 代表性原子问题

### 连接、约束和传力

- 删除或抑制加劲肋后，残留几何压印为何仍迫使部分钢梁从六面体改成四面体？
- 桥面和钢梁采用不同单元时，应使用共享拓扑、非共形接触、tie 还是 MPC？
- 预制梁与现浇桥面需要摩擦界面时，怎样既保持钢筋连续，又不把整个模型错误合并？
- 桥梁横向构件的 tie 警告是可忽略的微小调整，还是荷载路径已经错误？

### 模型层级和单元选择

- 支座系统中哪些零件可以先设为刚体，哪些必须保持柔性？
- 薄壁钢梁用三维实体时，厚度方向至少需要多少层单元才能得到可信弯曲应力？
- 何时应把系统模型切换成局部壳/实体模型，而不是继续细化全桥？
- 采用板单元分析偏心梁时，怎样恢复设计梁的组合内力？

### 验证和验收

- 实测与有限元自振频率相差约 50% 时，应先检查边界、材料、质量、附加质量还是网格？
- 高支座水平反力究竟来自斜交桥真实效应，还是支座方向、刚度和边界建模错误？
- 局部细化后某一点应力继续升高，如何区分有限应力集中、数学奇异性和后处理外推误差？
- 何种收敛证据足以支持工程签署，而不仅是“云图更平滑”？

### 网格布置与充分性

- 曲面或开孔几何在什么情况下必须通过分割和局部规则网格表达？
- 表面模型细节很多时，怎样保留较粗的全局网格，并只在线、点或局部面细化？
- 粗细网格过渡应离最终关注区多远？
- 0.1 m 板网格是否足以表达缺档引起的局部荷载路径，应该如何验证？

### 求解与自动化

- 长桥移动荷载应采用节点力时程、移动质量、接触还是专用移动荷载功能？
- 逐节点创建命名集和时程的流程怎样自动化？
- 接触不收敛时，应先判断物理定义、初始穿透、刚体运动、网格还是数值稳定化？
- 如何把一次成功模型固化为参数化、可追溯、可恢复的批量流程？

## 对 MeshPilot 的直接要求

### 1. 入口必须是工程问题，不是网格尺寸

系统首先收集：

- 要支持的工程决策；
- 构件、阶段、荷载和失效机制；
- 关注量 QoI、位置、单位和提取协议；
- 可用的解析解、规范、试验、现场数据和历史模型；
- 容许误差、时间、单元数、人工干预和审查等级。

### 2. 在给网格建议前，先做“模型可回答性”检查

至少检查：

- 梁/壳/实体/连接器是否能输出目标 QoI；
- 支座、边界和自由度是否符合真实构造；
- 构件是否连通并正确传力；
- 接触、粘结、滑移和分离是否被正确表达；
- 荷载、预应力和施工阶段是否映射到正确集合；
- 软化、开裂和失效模型是否具有网格客观性。

### 3. 网格策略必须由 QoI 路由

- 全局位移/内力：事件点、整体刚度和荷载分配；
- 疲劳/焊接细节：规定路径、板厚尺度和结构应力提取；
- 接触：接触斑、曲面和边缘压力梯度；
- 开裂/软化：特征长度、断裂能和局部化正则化；
- 屈曲：目标半波和缺陷表示；
- 动力/移动荷载：空间网格与时间步联合；
- 断裂：裂尖拓扑、轮廓稳定性和局部坐标。

### 4. 输出必须包含“为什么”和“怎样证明”

每条建议至少包括：

- 当前假设；
- 选择该模型和网格规则的原因；
- 可能失效的条件；
- 自动检查；
- 收敛/验证计划；
- 停止条件；
- 需要负责工程师确认的内容。

## 样本边界与下一轮

这不是概率抽样调查。英语软件社区占比高，一个帖子可贡献多个原子问题，频次只能解释为“在本样本中反复出现”。

下一轮应补充：

1. 桥梁设计院、检测评估、施工监控和软件支持工程师访谈；
2. 脱敏技术支持工单、模型审查意见和返工记录；
3. 中文工程社区、规范培训和设计院内部常见问题；
4. 对同一问题记录频率、严重度、工时损失和安全后果；
5. 让工程师对问题分类和 AI 所需输出进行复核。

当前仓库保留汇总、编码方法和代表性问题；未通过完整性校验的压缩明细已删除，不能作为可用数据。原子级明细需以可逐行审查的文本分片重新提交并通过 CI。
